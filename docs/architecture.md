# Architecture: Database Relations & System Flow

## Database Schema & Key Relations

- **Users**: core table with fields `id`, `name`, `matric_number`, `password`, `role` (enum: `Admin|Security|...`).
- **Items**: `id`, `user_id` (reporter), `category_id`, `type` (`Lost`/`Found`), `title`, `description`, `image_path`, `image_url`, timestamps.
- **Categories**: `id`, `name` — items belong to a category (`items.category_id → categories.id`).
- **Reports / Admin entities**: admin-managed reports are backed by `items` (or a dedicated `reports` table depending on future refactor) and reference `users` and `categories`.

### Relationships
- `users (1) ←→ (M) items` — a user can report many items.
- `categories (1) ←→ (M) items` — a category has many items.
- Admin actions reference `users` and `items` for management/auditing.

## Storage & Files
- Uploaded images are stored via `$request->file('image_file')->store(..., 'public')`.
- Public URLs are exposed using `Storage::url($path)`.
- Ensure `php artisan storage:link` is run so `public/storage` points to `storage/app/public`.
- Frontend uses the returned `image_url` for gallery display.

## System Flow (High-level)

### Student reporting flow
1. User fills Inertia/Vue form (`resources/js/Pages/Dashboard.vue`) and selects type (`Found`/`Lost`) and optionally `image_file`.
2. Client builds a `FormData` object and POSTs to the API endpoint (multipart/form-data).
3. Server `ItemController` / FormRequest validates input. Conditional rule used for image:
   - `Rule::requiredIf($this->type === 'Found')`.
4. Server stores the image on the `public` disk, saves `image_path` to DB, calculates `image_url` via `Storage::url($path)`, and returns item JSON or Inertia redirect.
5. Frontend refreshes gallery and displays the image using `image_url`.

### Admin management flow
1. Admin opens `/admin/login` and authenticates (`Admin\Auth\AuthenticatedSessionController`).
2. `EnsureUserRole` middleware enforces `Admin`/`Security` access for `/admin` routes.
3. Admin SPA (`resources/js/Pages/Admin/*`) calls `/admin/api/*` endpoints (`UsersController`, `ReportsController`).
4. Controllers use FormRequests for validation and Policies for authorization. Responses are JSON for SPA UX.
5. Admin triggers CRUD actions that update DB and the UI receives toast/notification updates.

## Routing & Endpoints
- Public/student:
  - `GET /dashboard` (Inertia page)
  - `POST /dashboard/data/items` (create item)
  - `PUT/PATCH /dashboard/data/items/{id}` (update)
  - `GET /api/items` (list)
- Auth:
  - `POST /login`, `POST /logout` handled by `AuthenticatedSessionController`
- Admin:
  - `GET /admin/login`, `POST /admin/login` (auth)
  - SPA pages at `/admin/*`
  - API: `/admin/api/users`, `/admin/api/reports` — CRUD endpoints returning JSON

## Validation, Auth & Security
- Validation centralized with FormRequests (`StoreReportRequest`, `UpdateReportRequest`, `StoreUserRequest`, etc.).
- Conditional image rule: `Rule::requiredIf($this->type === 'Found')`.
- Authorization enforced via Policies (`UserPolicy`, `ItemPolicy`) and `EnsureUserRole` middleware.
- Logout and account deletion invalidate the session and redirect to `/`.

## Frontend Build & Asset Flow
- Single Vite entry: `resources/js/app.js` (imports `../css/app.css`).
- Blade uses `@vite(['resources/js/app.js'])` and Vite writes `public/build/manifest.json`.
- Rebuild assets (`npm run build`) after changing entries so Inertia can resolve pages in production.
